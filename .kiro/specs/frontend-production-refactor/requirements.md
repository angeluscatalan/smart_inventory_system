# Requirements Document

## Introduction

This feature covers the production-readiness refactor of the InvSys PH Smart Inventory System frontend. The codebase was scaffolded with AI-assisted tooling and contains mock data imports, hardcoded credentials, placeholder handlers, and inline static data that must be removed before backend integration. The goal is a clean, professionally authored codebase where every interactive element has a documented API-ready stub, all mock data references are replaced with service-layer function stubs, and no AI-generation artifacts remain.

The project is a Next.js 14 application using TypeScript, Tailwind CSS, shadcn/ui, and Recharts. It covers eight dashboard pages (Dashboard, Inventory, Stock Operations, Branches, Notifications, Reports, User Management, Audit Logs) plus a Login page, with role-based access control for three roles: Admin, Branch Manager, and Staff.

## Glossary

- **Codebase**: All `.tsx` and `.ts` source files under `app/`, `components/`, and `lib/` directories.
- **AI Artifact**: Any comment, class name, HTML id, or text string that references an AI generation tool (e.g., "v0", "generated-by-ai", "ai-placeholder", "Powered by v0").
- **Mock Data Import**: Any `import` statement that references `@/lib/mock-data`.
- **Placeholder Handler**: An event handler that contains only a `console.log` call, an `alert()` call, or no implementation body.
- **API Stub**: A named, exported TypeScript function with a typed signature, a JSDoc comment describing its intended API contract, and a `// TODO: replace with API call` body.
- **Hardcoded Credential**: Any string literal representing a username or password embedded directly in component source code.
- **Static Chart Data**: An inline array literal defined inside a component file that is used as chart data and is not derived from props or a service call.
- **Naming Convention**: The project standard of PascalCase for React components and TypeScript interfaces, camelCase for functions and variables, and kebab-case for file names.
- **Service Layer**: A dedicated module (e.g., `lib/api/`) that encapsulates all data-fetching logic and exposes typed stub functions to components.
- **System**: The InvSys PH Smart Inventory System frontend application.

## Requirements

### Requirement 1: Remove AI Generation Artifacts

**User Story:** As a senior engineer, I want all AI-generation markers removed from the codebase, so that the code appears professionally authored and does not expose tooling metadata to reviewers or end users.

#### Acceptance Criteria

1. THE System SHALL contain no source file under `app/` or `components/` with a comment, class name, HTML `id`, or `data-*` attribute that includes the substrings `v0`, `generated-by-ai`, `ai-placeholder`, or `copilot-generated`.
2. THE System SHALL contain no rendered text string visible in the browser UI that contains any of the following: "Powered by v0", "Generated by AI", "AI Placeholder", "Copilot Generated", or any string that identifies the content as produced by an AI tool.
3. THE System SHALL contain zero case-insensitive matches for the literal patterns `v0`, `generated-by-ai`, `ai-placeholder`, or `copilot-generated` across all files in `app/` and `components/`.

---

### Requirement 2: Replace Mock Data Imports with Service Layer Stubs

**User Story:** As a backend engineer, I want all mock data references replaced with typed service stubs, so that I can implement real API calls by filling in the stub bodies without changing component interfaces.

#### Acceptance Criteria

1. THE System SHALL contain no `import` statement referencing `@/lib/mock-data` in any file under `app/` or `components/`.
2. THE System SHALL provide a `lib/api/` directory containing one service module per domain: `inventory.ts`, `branches.ts`, `users.ts`, `alerts.ts`, `activities.ts`, `stock-adjustments.ts`.
3. WHEN a component previously imported from `@/lib/mock-data`, THE System SHALL import the equivalent typed stub function from the corresponding `lib/api/` service module instead, without altering the component's prop interface.
4. THE System SHALL export from each service module at least one async stub function with a typed return signature and a JSDoc comment that describes the expected API endpoint, HTTP method, and response shape.
5. IF a service stub function is called before the real API is implemented, THEN THE System SHALL return a resolved `Promise` containing an empty array (`[]`) or a typed empty object that satisfies the declared return type at TypeScript compile time.

### Requirement 3: Replace Placeholder Event Handlers with Documented Stubs

**User Story:** As a backend engineer, I want every interactive UI element to call a named, documented stub function, so that I can locate and implement each handler without searching for anonymous or missing callbacks.

#### Acceptance Criteria

1. THE System SHALL contain no event handler in `app/` or `components/` whose function body contains only a `console.log()` call and no other statements.
2. THE System SHALL contain no event handler in `app/` or `components/` whose function body contains only an `alert()` call and no other statements.
3. WHEN the "Submit Adjustment" button in `StockAdjustmentForm` is activated, THE System SHALL call a named function `handleStockAdjustmentSubmit` that accepts a `StockAdjustmentPayload` parameter — where `StockAdjustmentPayload` has at minimum the fields `itemId: string`, `quantity: number`, `adjustmentType: 'add' | 'remove' | 'correction'`, and `reason: string` — and contains a `// TODO: replace with API call` comment.
4. WHEN the "Add Item" button in `InventoryPage` is activated, THE System SHALL call a named function `handleAddItem` that contains a `// TODO: open add-item modal or navigate to add-item form` comment.
5. WHEN the "Export" button in `InventoryPage` is activated, THE System SHALL call a named function `handleExportInventory` that contains a `// TODO: call export API and trigger file download` comment.
6. WHEN the "Add Branch" button in `BranchesPage` is activated, THE System SHALL call a named function `handleAddBranch` that contains a `// TODO: open add-branch modal` comment.
7. WHEN the "Edit" button in `BranchTable` is activated for a branch row, THE System SHALL call a named function `handleEditBranch` that accepts a `branchId: string` parameter and contains a `// TODO: open edit-branch modal` comment.
8. WHEN the "Delete" button in `BranchTable` is activated for a branch row, THE System SHALL call a named function `handleDeleteBranch` that accepts a `branchId: string` parameter and contains a `// TODO: call delete API with confirmation` comment.
9. WHEN the "Add User" button in `UserManagementPage` is activated, THE System SHALL call a named function `handleAddUser` that contains a `// TODO: open add-user modal` comment.
10. WHEN the "Edit" button in `UserManagementPage` is activated for a user row, THE System SHALL call a named function `handleEditUser` that accepts a `userId: string` parameter and contains a `// TODO: open edit-user modal` comment.
11. WHEN the "Delete" button in `UserManagementPage` is activated for a user row, THE System SHALL call a named function `handleDeleteUser` that accepts a `userId: string` parameter and contains a `// TODO: call delete API with confirmation` comment.
12. WHEN the "Generate Report" button in `ReportsPage` is activated, THE System SHALL call a named function `handleGenerateReport` that accepts a `ReportFilters` parameter — where `ReportFilters` has at minimum the fields `dateFrom: string`, `dateTo: string`, and `branchId: string | null` — and contains a `// TODO: call report generation API` comment.
13. WHEN the "Export to Excel" button in `ReportsPage` is activated, THE System SHALL call a named function `handleExportReport` that accepts a `ReportFilters` parameter (same shape as criterion 12) and contains a `// TODO: call export API and trigger file download` comment.
14. WHEN the global search `Input` in `Header` receives a change event, THE System SHALL call a named function `handleGlobalSearch` that accepts a `query: string` parameter and contains a `// TODO: implement global search` comment.
15. WHEN the "More Filters" button in `NotificationsPage` is activated, THE System SHALL call a named function `handleOpenNotificationFilters` that contains a `// TODO: open advanced filter panel` comment.
16. WHEN the "More Filters" button in `AuditLogsPage` is activated, THE System SHALL call a named function `handleOpenAuditFilters` that contains a `// TODO: open advanced filter panel` comment.
17. WHEN the "Export" button in `AuditLogsPage` is activated, THE System SHALL call a named function `handleExportAuditLogs` that contains a `// TODO: call export API and trigger file download` comment.
18. WHEN the "Profile" menu item in `Header` is activated, THE System SHALL call a named function `handleViewProfile` that contains a `// TODO: navigate to profile page` comment.

### Requirement 4: Remove Hardcoded Credentials

**User Story:** As a security-conscious engineer, I want all hardcoded usernames and passwords removed from the frontend source code, so that credentials are never exposed in the client bundle or version control history.

#### Acceptance Criteria

1. THE System SHALL contain no string literal matching any of the following demo passwords — `password123`, `manila2024`, `cebu2024`, `davao2024`, `staff123` — in any file under `components/`, `app/`, or `lib/` (excluding `lib/demo-accounts.ts`).
2. THE System SHALL contain no inline array or object that pairs a `username` field with a `password` field in any file under `components/` or `app/`.
3. WHERE a demo login panel is retained for development purposes, THE System SHALL load demo account data exclusively from `lib/demo-accounts.ts`, which SHALL be marked with a `// DEVELOPMENT ONLY — remove before production deployment` comment on its first line.
4. THE System SHALL contain no inline credentials record (pairing `username` and `password` fields) in `lib/auth-context.tsx` or any other file outside of `lib/demo-accounts.ts`; all such records SHALL be imported from `lib/demo-accounts.ts`.
5. WHEN the demo login panel is present in the rendered UI, THE System SHALL display a text label reading "Development Only" visually adjacent to the panel boundary, rendered as a non-interactive element.

### Requirement 5: Replace Static Inline Chart Data with Service Stubs

**User Story:** As a backend engineer, I want chart components to fetch data through service stubs rather than rendering hardcoded arrays, so that connecting real data requires only implementing the stub body.

#### Acceptance Criteria

1. THE System SHALL contain no inline array literal used as chart data defined inside `StockMovementChart`, `InventoryByBranchChart`, or `ExpiringProductsTimeline`.
2. THE System SHALL export `fetchStockMovementData`, `fetchInventoryByBranch`, and `fetchExpirationTimeline` as named async stub functions from `lib/api/stock-adjustments.ts` and `lib/api/inventory.ts` respectively, each with explicit TypeScript return types.
3. WHEN `StockMovementChart` mounts, THE System SHALL call `fetchStockMovementData()` from `lib/api/stock-adjustments.ts` to obtain chart data.
4. WHEN `InventoryByBranchChart` mounts, THE System SHALL call `fetchInventoryByBranch()` from `lib/api/inventory.ts` to obtain chart data.
5. WHEN `ExpiringProductsTimeline` mounts, THE System SHALL call `fetchExpirationTimeline()` from `lib/api/inventory.ts` to obtain chart data.
6. WHILE a chart data fetch is in progress, THE System SHALL render a loading skeleton or spinner within the chart card area in place of the chart content.
7. IF a chart data fetch returns an error, THEN THE System SHALL render an inline error message within the chart card area, replacing the chart content, and SHALL NOT render a toast notification or modal for this error.

### Requirement 6: Replace Hardcoded Audit Log Statistics with Computed Stubs

**User Story:** As a backend engineer, I want the audit log page statistics (Last 24 Hours, Last Week, Active Users) to be derived from a service stub rather than hardcoded literals, so that the values reflect real data once the API is connected.

#### Acceptance Criteria

1. THE System SHALL contain no numeric literal used as a static display value for "Last 24 Hours", "Last Week", or "Active Users" statistics in `AuditLogsPage`.
2. WHEN `AuditLogsPage` mounts, THE System SHALL call `fetchAuditLogStats()` from `lib/api/activities.ts`, where `fetchAuditLogStats` returns `Promise<AuditLogStats>` and `AuditLogStats` is a named type with at minimum the fields `last24Hours: number`, `lastWeek: number`, and `activeUsers: number`.
3. WHILE `fetchAuditLogStats()` is in progress, THE System SHALL display a loading indicator (skeleton or spinner) in place of each statistic value.
4. IF `fetchAuditLogStats()` rejects or throws, THEN THE System SHALL display the string `--` in place of each statistic value, and SHALL NOT display `--` when the call resolves successfully.

### Requirement 7: Standardize Naming Conventions

**User Story:** As a developer, I want the codebase to follow consistent naming conventions throughout, so that the code is readable and maintainable without requiring knowledge of its generation history.

#### Acceptance Criteria

1. THE System SHALL name all React component functions and TypeScript interfaces using PascalCase.
2. THE System SHALL name all non-component functions, variables, and React hooks using camelCase.
3. THE System SHALL name all source files under `app/`, `components/`, `hooks/`, and `lib/` using kebab-case.
4. THE System SHALL name all TypeScript type aliases and interfaces with a noun or noun phrase that does not include the prefix `I` (e.g., `BranchTableProps` not `IBranchTableProps`).
5. THE System SHALL use the suffix `Props` for all React component prop interfaces (e.g., `InventoryTableProps`, `AlertListProps`).
6. THE System SHALL use the suffix `Payload` for interfaces representing data submitted to an API (e.g., `StockAdjustmentPayload`), and the suffix `Filters` for interfaces representing query or filter parameters (e.g., `ReportFilters`).

---

### Requirement 8: Ensure All Stub Functions Are Typed

**User Story:** As a TypeScript developer, I want every stub function to have explicit parameter and return types, so that the compiler enforces the contract before the real implementation is written.

#### Acceptance Criteria

1. THE System SHALL declare all API stub functions in `lib/api/` with explicit TypeScript parameter types and explicit return types.
2. THE System SHALL declare all named event handler stubs listed in Requirement 3 (e.g., `handleStockAdjustmentSubmit`, `handleAddItem`, `handleExportInventory`, etc.) with explicit TypeScript parameter types and explicit return types.
3. THE System SHALL not use the `any` type in any stub function signature in `lib/api/` or in any named event handler stub in `app/` or `components/`.
4. IF an API stub function in `lib/api/` has no parameters, THEN THE System SHALL declare it as `(): Promise<ReturnType>` where `ReturnType` is a named type or an array of a named type exported from `lib/types.ts`.
